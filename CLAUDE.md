# finagent — Claude Code Context

## What this is

Personal financial intelligence agent for a household of 2-3 users (Alice + partner). Answers natural-language questions about banking transactions, spending, and recurring payments via a ReAct agent loop backed by any OpenAI API-compatible LLM.

Phase 1 scope: banking schema + agentic loop skeleton. Investments, tagging pipeline, embeddings, external data providers, and tax assistance are deferred to Phase 2+.

## Prerequisites

- Go 1.26+
- Docker (for PostgreSQL 18 + pgvector via docker-compose)
- `DATABASE_URL` env var pointing to the running Postgres instance

## Build & Run

```bash
# Start database
docker compose up -d

# Run all migrations
make migrate-up

# Generate sqlc types from queries
make generate

# Format code
make fmt

# Build binary
make build

# Run CLI
./bin/finagent --user alice
# or
make run
```

## Architecture

```
cmd/finagent/          — main wiring: config → stores → tools → agent → channel
config/                — koanf config loading (YAML + ENV override)
internal/
  agent/               — ReAct loop (max 8 tool rounds), system prompt, model router
  api/                 — gorilla/mux HTTP server: POST /api/chat, GET /api/health
  channel/             — Channel + Message + Response interfaces
    cli/               — readline CLI; user from --user flag
  llm/                 — Provider interface + all shared types
    openai/            — OpenAI API-compatible client (go-openai)
  store/               — Thin wrappers over sqlc-generated code
  tools/               — Tool registry + individual tool implementations
  db/
    migrations/        — golang-migrate SQL files (embedded in binary)
    queries/           — sqlc query files
  sqlc/                — AUTO-GENERATED by sqlc — never hand-edit
  model/               — Money type (paise), shared error types
```

## Key Conventions

**Money**: always stored as `BIGINT` paise (INR × 100). No floats anywhere. `model.Money` wraps int64.

**Transactions are immutable**: `transactions` table stores exactly what the bank reported — never modified after insert. All agent enrichment (category, notes, labels, transfer links) lives in `transaction_enrichments`.

**Idempotency key**: `SHA256(account_id || txn_date || amount || description)`. Import pipelines use this to skip duplicates.

**account_class** is a PostgreSQL generated column computed from `account_type` — it is never set manually.

**counterparty_identifier** (VPA like `zomato@axisbank`, or account+IFSC) is the stable merchant identity key across banks — not the description string.

**transfer_id** is a shared UUID on `transaction_enrichments` that groups related records: own-account transfers, household transfers, or bank→investment links. `transfer_type` on the same row says which kind.

**Categories vs Labels**: categories are system-defined, direction-constrained, two-level hierarchy (depth 0=category, 1=sub-category). Labels are flat, user-defined, no direction constraint.

**User scoping**: every tool scopes by `user_id` by default. Tools accept an optional `user_id` argument to query another household member's data.

## Go Conventions

- Package names: lowercase, single word
- Every exported symbol gets a godoc comment
- Errors wrapped: `fmt.Errorf("list transactions: %w", err)`
- `context.Context` always first parameter
- Interfaces defined in the consuming package, not the implementing package
- No global state — all dependencies injected via `New*(...)` constructors
- Modern Go 1.26 idioms: `any`, `new(val)`, `errors.AsType[T]`, `wg.Go`, `slices.*`, `maps.*`, `cmp.Or`, `omitzero`, `for i := range n`
- Run `make fmt` before every commit

## sqlc

Queries live in `internal/db/queries/`. Run `make generate` after editing queries or schema. The `internal/sqlc/` directory is auto-generated — never edit files there directly.

## Migrations

`internal/db/migrations/` contains numbered SQL files. They are embedded into the binary via `//go:embed`. Auto-migration on startup is controlled by `database.auto_migrate` in config.

Migration order:
1. extensions + enums
2. users
3. accounts, account_members, account_details
4. categories, labels
5. transactions, transaction_enrichments, transaction_labels, transaction_splits, transaction_split_labels
6. recurring_payments + FK on transaction_enrichments
7. agent_memories
8. conversation_sessions, conversation_messages
9. import_runs
10. v_transactions view
11. seed system categories
